## 2. Структура кода — как объяснить за 2 минуты

**RU :** Используйте эту схему устно:

> «Приложение состоит из трёх слоёв.
> **Слой данных** — это список из 118 словарей Python с химическими свойствами
> и несколько вспомогательных словарей с данными вычислительной химии.
> **Слой интерфейса** — это `app.layout`, который описывает все компоненты Dash:
> выпадающие меню, график, панель деталей, кнопки языка.
> **Слой логики** — четыре callback-функции, которые Dash вызывает автоматически
> при взаимодействии пользователя с интерфейсом.»

Нарисуйте на доске (или покажите диаграмму из презентации):

```
ДАННЫЕ (Python-структуры)
        ↓
    LAYOUT (app.layout)
        ↕  callback
   CALLBACKS (4 функции)
        ↕  JSON / HTTP
      БРАУЗЕР
```

**FR :** Utilisez ce schéma en trois couches : Données → Layout → Callbacks → Navigateur.

## 3. Вероятные вопросы по коду и ответы

---

### «Что такое callback в Dash?»

**RU :**
> «Callback — это Python-функция, декорированная `@app.callback`. Декоратор
> принимает `Output` (что обновить), `Input` (что слушать) и опционально `State`
> (что читать без запуска callback). Когда значение `Input` меняется в браузере,
> Dash отправляет POST-запрос на сервер, вызывает функцию и возвращает результат
> в виде JSON. Браузер применяет обновление без перезагрузки страницы.»

**Пример из кода:**
```python
@app.callback(
    Output("periodic-figure", "figure"),   # что обновить
    Input("block-filter", "value"),        # что слушать
    Input("lang", "data"),                 # тоже Input
)
def update_graph(block_value, lang):
    return build_periodic_figure(block_filter=block_value, lang=lang)
```

---

### «Почему вы использовали `shapes` вместо маркеров Plotly для ячеек?»

**RU :**
> «Маркеры Plotly имеют размер в пикселях. Если соотношение сторон графика
> меняется (разное разрешение экрана, изменение размера окна), маркеры
> становятся прямоугольными, а не квадратными. `layout.shapes` задаётся
> в координатах данных (`col ± 0.46`, `row ± 0.46`), поэтому ячейки остаются
> квадратными всегда. В сочетании с `scaleanchor="x", scaleratio=1` на оси Y
> это гарантирует идеальную сетку при любом разрешении.»

---

### «Что такое `customdata` и зачем оно нужно?»

**RU :**
> «`customdata` — это механизм Plotly для прикрепления произвольных данных
> к каждой точке графика. Я передаю 20 полей на элемент (атомный номер,
> символ, название, данные вычхим…). Когда пользователь кликает по ячейке,
> Dash передаёт в callback массив `customdata` этой точки через `clickData`.
> Это намного быстрее, чем делать поиск по `atomic_number` в DataFrame
> на каждый клик.»

**Индексы в коде:**
```
[0]=Z  [1]=символ  [2]=название(FR)  [5]=блок
[6]=ECP  [7]=basis  [12]=IE₁  [19]=название(RU)
```

---

### «Как работает переключение языка?»

**RU :**
> «Я использую `dcc.Store` — компонент Dash, который хранит данные в браузере
> без отображения. Store содержит строку `"fr"` или `"ru"`. Кнопки FR/RU
> запускают callback `toggle_lang`, который записывает новое значение в Store.
> Это изменение Store автоматически запускает два других callback:
> `update_ui_language` (переводит все тексты интерфейса) и `update_graph`
> (перестраивает инфоблок над элементами). Таким образом, один клик
> по кнопке вызывает цепочку из трёх обновлений без перезагрузки страницы.»

---

### «Что такое `ctx.triggered_id`?»

**RU :**
> «Когда несколько кнопок могут запустить один и тот же callback (у меня FR
> и RU запускают `toggle_lang`), нужно знать, какая именно кнопка была нажата.
> `ctx.triggered_id` возвращает строку с `id` компонента, который вызвал
> callback. Это доступно в Dash начиная с версии 2.4 и гораздо чище, чем
> разбор `ctx.triggered_prop_ids`.»

---

### «Почему лантаниды и актиниды расположены отдельно?»

**RU :**
> «В стандартном формате таблицы Менделеева лантаниды (57–71) и актиниды
> (89–103) выносятся в отдельные строки под основной таблицей, чтобы она
> не была слишком широкой. Я реализовал это, добавив дублирующие записи
> для La (Z=57) и Ac (Z=89) с `row=8` и `row=9` соответственно. Исходные
> записи с `row=6` и `row=7` показывают ячейку-заглушку в основной таблице.»

---

### «Почему данные хранятся в Python-словарях, а не в CSV или базе данных?»

**RU :**
> «Это учебный проект с фиксированным набором данных — 118 элементов,
> которые не меняются. Хранить данные в Python-структурах внутри `app.py`
> делает приложение полностью автономным: не нужно читать файл, не нужна
> база данных, нет зависимости от файловой системы. Это упрощает деплой —
> достаточно одного `.py`-файла и `requirements.txt`.»

## 4. Вопросы по CSS и ответы

---

### «Как вы реализовали адаптивную вёрстку?»

**RU :**
> «Я использовал CSS Grid. Основная сетка приложения задаётся как
> `grid-template-columns: 2fr 1fr` — таблица занимает две трети ширины,
> панель деталей — одну треть. Блок фильтров — `repeat(4, 1fr)` — четыре
> равные колонки. Через `@media`-запросы при ширине менее 1 024 px
> обе сетки переключаются на одну колонку, чтобы интерфейс оставался
> читаемым на планшете.»

---

### «Почему вы использовали `!important` в CSS для выпадающих меню?»

**RU :**
> «Dash генерирует собственные инлайн-стили для компонента `dcc.Dropdown`
> через React. Эти инлайн-стили имеют более высокую специфичность,
> чем любой внешний класс. Без `!important` моя тёмная тема игнорировалась,
> и текст выпадающего списка оставался чёрным на тёмном фоне — невидимым.
> Это известное ограничение Dash v2 при кастомизации Dropdown.»

---

### «Что означает `rgba(255,255,255,0.14)` у затемнённых элементов?»

**RU :**
> «Я не меняю цвет ячейки при фильтрации — я меняю параметр `opacity`
> в `layout.shapes` Plotly. Значение `0.14` означает 14% непрозрачности:
> ячейка почти прозрачна, но структура таблицы остаётся видимой.
> Это лучше, чем полное скрытие элементов, потому что пользователь
> не теряет контекст расположения.»

  чтобы объяснять логику преподавателю. Прочитайте один-два вслух при объяснении.

  чем пролистать весь файл за 30 секунд.
